# ADR-016: Database Migration Strategy

## Status

Accepted

_Implemented via `alembic.ini` + `collector/db/migrations/`, Phase 2._

## Context

Phase 2 introduces the Collector's first real database schema (`nodes`,
`metric_samples`, per `docs/adr/002-postgresql-choice.md`). Phase 0 already flagged this
decision as needed "as soon as `collector/db/models` exists"
(`docs/architecture/00-project-initialization.md` §3.1) — that condition is now true, and
this ADR resolves it in the version-controlled record, not just a design-doc footnote.

## Decision

Adopt Alembic for schema evolution. The Collector's tables are created and changed only
through versioned migration scripts (`collector/db/migrations/versions/`), applied via
`alembic upgrade head` as an explicit deployment step — **never** via
`Base.metadata.create_all()` at application startup. The initial migration
(`0001_initial_schema.py`) was hand-written rather than autogenerated, since no live
database was reachable in the development environment to diff against; it was verified by
generating its offline SQL (`alembic upgrade head --sql`) and inspecting the resulting DDL
against the ORM model definitions.

## Alternatives Considered

- **`Base.metadata.create_all()` at app startup** — zero setup, works immediately.
  Rejected: no history of schema changes, no rollback path, and no safe way to evolve a
  schema that already has production data — exactly the "tutorial project" shortcut
  `.claude/CLAUDE.md` explicitly says to avoid. It also can't express a change like
  "rename a column" or "backfill a new non-nullable column" — only "create if missing."
- **Auto-migrate on app boot** (run `alembic upgrade head` programmatically inside
  `create_app()` or a startup event) — convenient for a single-instance dev setup, but a
  known anti-pattern once more than one Collector instance can start concurrently (two
  processes racing to apply the same migration). Rejected in favor of migrations being an
  explicit, operator-triggered deployment step (documented in `collector/README.md`).
- **Autogenerating the initial migration against a live Postgres** — the more typical
  Alembic workflow, and preferred when a database is available. Not possible in this
  environment (Docker daemon not running — confirmed via `docker ps` before starting
  Phase 2 implementation); hand-writing plus offline-SQL verification was the available
  substitute, not the first choice.

## Consequences

- Every future schema change is a new migration file, reviewed like any other code
  change — more process than `create_all()`, but the correct amount for a system
  `.claude/CLAUDE.md` explicitly wants to resemble production software.
- `collector/db/migrations/env.py` reads its database URL from `CollectorSettings`
  (i.e., from the same environment variables the running app uses) rather than a
  hardcoded value in `alembic.ini` — one source of truth for "where is the database,"
  avoiding drift between what the app connects to and what migrations are applied against.

## Update: verified against real PostgreSQL (post-Phase-4)

All three migrations (`0001`, `0002`, `0003`) were later applied for the first time
against a real PostgreSQL 16 container (`alembic upgrade head` — not just `--sql`), and
the Collector was run against it end-to-end (ingestion, rule evaluation, alert
open/resolve/acknowledge, all confirmed via direct `psql` inspection). This closed the gap
this ADR originally flagged: "verified the migration is well-formed" now also means
"verified it applies cleanly to a real database and the app works against it."

That verification **found a real, previously-hidden bug**: SQLAlchemy's `Enum(SomeEnumClass)`
column type defaults to storing each member's `.name` (e.g. `"CPU_USAGE_PERCENT"`), not
its `.value` (e.g. `"cpu.usage_percent"`) — but every migration's hand-written
`sa.Enum(*_VALUES, name=...)` defines the Postgres enum type's labels from `.value`. Every
enum-typed column (`metric_type`, `rule_kind`, `severity`, `status`) had this mismatch.
It never surfaced against SQLite because `Base.metadata.create_all()` (used only by the
test suite, never production) derives its own CHECK constraint from the same
(equally-wrong) default — self-consistent, and therefore invisible, until compared
against the independently-authored migration DDL that a real Postgres actually enforces.
Fixed via `collector/db/enum_column.py`'s `str_enum_column()` helper
(`values_callable` set to extract `.value`), applied to all four columns, with a
regression test (`tests/unit/collector/test_db_enum_column.py`) that asserts against the
*raw stored text* so this specific regression can't hide behind SQLite again.

This is the clearest demonstration yet of this ADR's own "Interview Talking Points"
distinction between "verified well-formed" and "verified against a real database" — the
former genuinely did not catch this, and only the latter did.

## Interview Talking Points

The interesting constraint here wasn't "Alembic vs. nothing" — that's a foregone
conclusion for a real system — it was building and validating a real migration without a
database to run it against. Offline-mode SQL generation (`alembic upgrade head --sql`) is
exactly the tool Alembic provides for this: it renders the DDL Alembic *would* execute
without needing a live connection, which is enough to catch authoring mistakes (wrong
column type, missing FK, bad enum values) even though it can't catch runtime surprises
(permissions, existing conflicting objects) that only a real target database would surface.
That distinction — "verified the migration is well-formed" vs. "verified it applies
cleanly to a real database" — is the honest boundary of what got tested in this phase.
